## Top Stories

### 1. Seven Things I've Learned Getting Companies to Use AI
**Source:** Every (Also True for Humans)

Mike Taylor, head of tech consulting at Every, shares a battle-tested playbook from the AI consulting front lines. The core insight: mandating AI use backfires — instead, find internal champions and give them air cover. His most counterintuitive advice is to assign "impossible tasks": set goals that literally cannot be achieved without AI (e.g., one blog post per day vs. weekly), then ask employees what they need to get there. The most impactful individual habit is mandatory AI note-taking (Granola + MCP connector), which Taylor says unlocks 80–90% of AI's value by extracting structured information from unstructured meetings. The deeper shift: every individual contributor is now an agent manager without management training — they need to learn context switching, system design, and output evaluation.

### 2. Stripe Built AI "Minions" That Ship 1,300 Pull Requests Per Week
**Source:** Lenny's Newsletter (How I AI podcast)

Stripe engineer Steve Kaliski reveals how Stripe's AI coding agents ship roughly 1,300 PRs weekly, triggered by a single Slack emoji. The key enablers: years of investment in developer experience (clear docs, blessed paths for common tasks) that now directly translate to higher agent success rates; cloud-based dev environments that spin up in seconds for parallel agent work; and automated confidence signals (test coverage, synthetic e2e tests, blue-green deployments) that let humans review AI-generated code at scale. The bottleneck is no longer writing code — it's having enough good ideas. A live demo showed Claude Code autonomously paying third-party services (BrowserBase, Parallel AI) to complete tasks, pointing toward a future of ephemeral "hyper-useful single API" businesses optimized for agent consumers rather than humans.

### 3. How to Evaluate AI Fluency in Technical Interviews
**Source:** Engineering Leadership (Gregor Ojstersek / Hamid Moosavian)

Banning AI from technical interviews is outdated — 50%+ of developers use AI daily, and the goal should be measuring *how well* candidates use it. Guest author Hamid Moosavian proposes a four-level AI fluency framework: Avoidant (refuses AI entirely) → Dependent (accepts output blindly, can't debug it) → Competent (critical review, catches errors, tests output) → Advanced (knows when *not* to use AI, treats it as a thought partner). The practical differentiator: Level 2/3 engineers write context-rich prompts with constraints and file paths, integrate incrementally, and check generated packages for existence and security issues. Companies like Meta and Canva have already redesigned their interview processes to include AI — the competitive signal is now whether candidates can produce maintainable code *with* AI, not whether they avoid it.

Leaked Coatue investor slides from January 2026 reveal the firm projects Anthropic at a $1.995 trillion valuation by 2030 and $2.413 trillion by 2031, based on $200 billion in revenue and $48 billion EBITDA by 2031 — a 41× forward EBITDA multiple. ### 5. How Roblox Translates 16 Languages in 100 Milliseconds
**Source:** ByteByteGo

Roblox built a single unified translation model for all 256 language pairs (not 256 separate models) serving 70 million daily users across 180 countries at ~100ms latency and 5,000+ chats/second. The architecture uses Mixture of Experts (activating only relevant subnetworks per language pair), knowledge distillation to compress a 1-billion-parameter teacher to a 650-million-parameter student, and two caching layers — one for exact-match translations and one for intermediate embeddings (so a Korean message going to English, German, and French speakers only runs the encoder once). The engineering challenge wasn't building a model that could translate — it was building infrastructure that could do it fast enough for a real-time conversation. Low-resource language pairs (e.g., French-Thai) remain the weak link, addressed through iterative back-translation.

### 6. Vibe Coding SwiftUI Apps: Simon Willison Builds Two macOS Tools with Zero SwiftUI Knowledge
**Source:** Simon Willison

Simon Willison built two production-quality macOS menu bar apps using Claude Opus 4.6 — Bandwidther (per-app network bandwidth monitor) and Gpuer (GPU and unified memory dashboard for Apple Silicon) — without opening Xcode or writing a line of Swift himself. Total prompts: roughly 8–10 per app. His most reusable technique: point the agent at an already-built app to "recombine elements" — the second app was built by telling Claude to look at Bandwidther and imitate its menu bar pattern. Both apps are single-file SwiftUI, published as open source. This is a concrete data point on vibe coding's current ceiling: it works well for self-contained desktop utilities with well-documented frameworks, less so for complex production systems.
