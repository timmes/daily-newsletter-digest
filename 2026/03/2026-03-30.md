# Daily Newsletter Digest - 2026-03-30

## Top Stories

### 1. How to Evaluate AI Fluency in Technical Interviews — *Engineering Leadership (Gregor Ojstersek)*
AI-resistant interviews are now outdated: over 97% of developers plan to use AI tools, yet most hiring processes still penalise candidates for using them. Guest author Hamid Moosavian introduces a four-level AI fluency framework — from Level 0 (AI-avoidant) through Level 3 (AI-advanced, knows *when not to* use AI). Companies like Meta, Canva, Goldman Sachs, and Intuit have already shifted: they allow AI tools in interviews and observe how candidates use them. The evaluation axis moves from "does it work?" to "how was it built, and why?" Context-rich prompting, incremental integration, and critical review of output separate Level 2–3 engineers from those who blindly accept AI suggestions and produce unmaintainable code.

### 2. Plus One: Every Launches One-Click AI Coworkers — *Every (Dan Shipper)*
Every is launching "Plus Ones" — hosted OpenClaw agents that live in Slack, pre-loaded with Every's full tool stack. One-click setup, no dedicated machine needed (previous OpenClaw users had to run Mac Minis 24/7). Pre-loaded skills include content digest, daily brief, animation generation, and frontend design. Agents arrive pre-connected to Cora (email), Spiral (writing voice), Proof (live docs), Google Workspace, and Notion. Every's team already runs a "parallel org chart" of named AI coworkers — Iris handles product marketing ops, Alfredo handles visual design, R2-C2 triages bugs and co-authors articles. Rolling out at 20 per week, Every subscribers first.

### 3. Good Judgment Is the New Important Skill for Engineers — *Engineering Leadership (Gregor Ojstersek)*
As AI commoditises code generation, the question shifts from "can you build it?" to "should you build it, and how?" Good engineering judgment now means: knowing when a feature shouldn't be built, spotting over-engineered solutions, forecasting the long-term cost of shortcuts, and balancing speed against quality. Analogy: the Printing Press made books abundant, so the skill shifted from accessing knowledge to curating it — same shift is happening with code. Judgment matters most in smaller companies with high ownership (e.g. OpenAI's bottom-up culture). Frameworks for developing it: reflective practice, seeking feedback, understanding trade-offs, internalising the "why" behind decisions.

### 4. Is GitHub Still the Top Platform for AI-Native Development? — *The Pragmatic Engineer (Gergely Orosz)*
GitHub has dropped to roughly one nine (~90%) availability because AI agents are overwhelming its infrastructure — Claude Code's load on GitHub has 6×'d in three months. GitHub has no CEO (Thomas Dohmke stepped down; no replacement hired), is entangled in Microsoft's Copilot politics, and has no clear "North Star" around AI-native git hosting. Startup Pierre Computer claims 15,000 repos/minute versus GitHub's 230/min. GitHub Copilot is losing the agentic coding narrative to Claude Code and Cursor. The Pragmatic Engineer argues GitHub urgently needs to choose: become critical AI agent infrastructure, or keep shipping incremental features into irrelevance.

### 5. Build Your Own Bloomberg Terminal With AI — *Every (Brooker Belcourt)*
Former hedge fund analyst walks through four levels of AI-powered investment research — from a basic ChatGPT project with uploaded SEC filings to a fully automated overnight pipeline that reads your financial model, applies your investment thesis, and runs while you sleep. Most finance teams quit AI after one bad ChatGPT result, not realising they're judging the whole technology on the least sophisticated tool. The guide shows how structured data connectors, correct prompting, and iterative model refinement close the gap between basic summarisation and actual analyst-grade output. Level four: "four words and the whole workflow runs."

### 6. Vibe Coding SwiftUI Apps Is a Lot of Fun — *Simon Willison*
Simon Willison built two production-quality macOS monitoring apps (Bandwidther: network bandwidth per process; Gpuer: GPU/RAM usage) using Claude Opus 4.6 with zero prior SwiftUI knowledge. A full SwiftUI app fits in a single file (~800–1,000 lines), and Claude Code's design taste for macOS UI is surprisingly good — menu bar apps add just a few extra lines. Key technique: the "recombine" trick — pointing Claude at features in one in-progress project and asking it to replicate them in another. Important caveat: Simon warns he can't verify whether the numbers these apps report are accurate, as he lacks macOS internals knowledge. Both repos are open source.
