# Daily Newsletter Digest - 2026-04-07

## Top Stories

### 1. How to Design for Human-Agent Interaction — *Every / Thesis* (Karri Saarinen, Linear CEO)
Linear CEO Karri Saarinen argues that AI's unpredictability is primarily a design problem, not a model problem — and that solving it falls to product designers, not AI researchers. He presents six principles Linear has developed for human-agent collaboration: agents must always disclose they're agents (never impersonate humans), they should use the same native actions as humans (buttons, status updates, the same UI), they must give instant feedback when invoked, they must be transparent about their internal reasoning state, they must stop immediately and stay stopped when asked, and they cannot be held accountable — the human who delegated to them is. The key insight is that designing a well-structured environment before the agent runs (clear plans, issue backlogs, documented constraints) removes the need for constant human rubber-stamping mid-task. Saarinen's deepest claim: the "slippery feeling" people associate with AI products almost always traces back to the interface, not the underlying model. Linear, used by OpenAI and Coinbase to manage their own AI agents, is a rare case of a SaaS product that adapted to the agent era rather than being disrupted by it.

### 2. The Claude Code Leak Showed Me What I Was Configuring Wrong — *The AI Architect* (Tyler Folkman)
When Claude Code's full source code leaked via a sourcemap file last week, Tyler Folkman spent days reading it and found that most users — himself included — were misconfiguring the tool in the same predictable ways. The biggest mistake: writing rules in CLAUDE.md when hooks are available. CLAUDE.md is probabilistic (the model may forget mid-session), while hooks are deterministic — a PreToolUse hook that blocks `rm -rf` fires every single time, even in YOLO mode. A second insight: Claude Code's memory system enforces a strict 4-type taxonomy (user, feedback, project, reference) and explicitly excludes anything derivable from reading the codebase (file paths, git history, debugging patterns) — saving these wastes the 5 memory-selection slots per query. Third, CLAUDE.md is an attention budget: each line competes with system-prompt instructions, so files over 150 lines see per-rule adherence collapse. Fourth, skill frontmatter (not body content) is all the model sees at discovery time — skills with vague names and no `when_to_use` field never get invoked. Fifth, each MCP server can cost 15,000+ tokens in context just for tool definitions. Folkman made five specific configuration changes after reading the source, each taking under 30 minutes.

### 3. How to Use OpenClaw as an Engineering Leader — *Engineering Leadership* (Gregor Ojstersek)
Gregor Ojstersek walks through what OpenClaw (formerly ClawdBot, renamed after Anthropic trademark complaints, then acquired by OpenAI in February) actually is and how engineering leaders can use it — not as a chatbot, but as an autonomous assistant that breaks down tasks, spins up sub-agents, and delivers outcomes instead of ideas. The article covers the full setup: installing on a dedicated Mac mini (not your main machine — OpenClaw gets full system access), wiring up Telegram as the messaging interface, setting up multiple API keys (Anthropic + OpenAI/Gemini) to avoid rate limits, and using the gateway dashboard at localhost:18789 to monitor activity. Ojstersek completed three engineering projects using it and identifies the highest-value use cases for engineering leaders: code review workflows, architectural documentation, and coordinating multi-step technical tasks across teams. His honest caveat: judgment remains more important than ever as AI commoditises implementation — the risk is building a "Jenga tower codebase" when you approve AI-generated code without fully understanding it. OpenClaw now has ~350k GitHub stars and over 46k skills published in its Clawhub marketplace.

### 4. Our First OpenClaw Teammate — *Pirate Skills* (Ben Sufiani)
Pirate Skills gave their first AI agent, ChristAIna (their "Quarter Master"), a Slack workspace, tools, and a list of responsibilities — and she started acting like a colleague, not an assistant. Running on a Hetzner server costing ~$12/month with Claude Opus 4.6 as her brain, ChristAIna operates across seven Slack channels: morning pipeline briefings (#brief), code deploys (#develop), CRM pipeline checks with follow-up reminders in Attio (#crm), content drafts and research (#content), and tax/scheduling/logistics (#orga). The full behind-the-scenes article covers the setup, real use cases, what still breaks, and a Ship Plan for teams wanting to replicate it. Sufiani's key insight is that the leap from AI assistant to AI teammate is not about smarter models — it's about infrastructure and onboarding, the same investment you'd make for a human hire: a workspace, communication channels, tool access, and clear responsibilities. Claude Code did 95% of the setup; Sufiani created accounts and pasted tokens.

### 8. House Rules for the Agents — *Every / Context Window*
Every's Context Window team published a comprehensive guide to CLAUDE.md configuration for agentic workflows, framing it as "house rules" that govern how Claude Code behaves inside your project. The core insight, consistent with the leaked source code analysis: CLAUDE.md is read probabilistically, so it works best for preferences and defaults rather than hard rules (which belong in hooks). The guide distinguishes between rules that should be in CLAUDE.md (coding style preferences, project-specific vocabulary, default approaches) versus rules that should be hooks (anything that must always happen: formatting, blocking dangerous commands, running tests after edits). A practical example: instead of writing "always run the tests after making changes" in CLAUDE.md — which Claude might forget — wire a PostToolUse hook that fires `npm test` automatically. The guide also covers how context windows fill up fast in long agentic sessions, and why compaction matters for maintaining coherent multi-step task execution.

### 9. Writing With AI is Harder Than You Think — *Every / Working Overtime* (Katie Parrott)
Katie Parrott argues that AI-assisted writing requires more rigor, judgment, and willingness to revise than most people expect — not less. The popular assumption is that AI lowers the bar and speeds things up, but in practice it surfaces two kinds of failure: over-reliance (accepting AI output without critical evaluation, producing generic work) and under-use (treating AI as a spell-checker rather than a thinking partner). Good AI-assisted writing means being willing to tell the AI — and yourself — that the draft isn't good enough yet. Parrott's framework: use AI for first drafts, structure, and research synthesis, but maintain full editorial control at the revision stage. The writers who produce the best AI-assisted work are those who already had strong editorial instincts before AI arrived — the tool amplifies taste, it doesn't substitute for it.

### 10. Measuring AI Impact, Assessing Readiness & New Data Trends — *Engineering Enablement* (Abi Noda)
Engineering Enablement's latest issue covers three frameworks for understanding AI's actual impact on engineering teams. On measurement: AI-authored merged code has plateaued at ~27-30% across their tracked organizations, and the bottleneck is shifting from writing code to reviewing it — teams need to rethink how they measure developer productivity when the human's primary value is now judgment and review capacity rather than output volume. On readiness: the "amplification vs. augmentation" framework helps teams distinguish between AI making existing engineers faster (amplification) versus enabling engineers to take on work they couldn't do before (augmentation) — both matter, but they require different investments. On trends: background agents (Claude Code, Devin, etc.) are making humans the bottleneck in ways that existing DevOps metrics don't capture — daily active users of AI coding tools lead adoption, with power users getting disproportionate gains.
