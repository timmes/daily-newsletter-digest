### 1. Claude Code's Source Leak: 5 Things You Were Configuring Wrong
*The AI Architect — Tyler Folkman*

Claude Code's full source code leaked this week, and Tyler Folkman spent days reading through it. The biggest finding: most users are leaving half the capability on the table. The most common misconfiguration is putting everything in CLAUDE.md — which is **probabilistic** (the model reads it and tries to follow it, sometimes forgetting mid-conversation). Hooks are **deterministic**: a PreToolUse hook blocking `rm -rf` will always block it, every time, even in YOLO mode. The rule: if it must always happen, make it a hook. If it should usually happen, put it in CLAUDE.md. Other changes: (2) The leaked memory system enforces a strict 4-type taxonomy (user, feedback, project, reference) and explicitly excludes anything derivable from code — file paths, git history, debugging recipes waste your 5-selection-slot budget. (3) CLAUDE.md is a **budget** of ~100 prescriptive constraints, not a document — audit every line, delete anything derivable. (4) Skills that don't have a `when_to_use` frontmatter field never get auto-invoked — the model only sees frontmatter at session start, not the body. (5) Each MCP server can inject 15,000+ tokens into your context; 5 servers = 60,000 tokens before you type anything.

### 2. Measuring AI Impact Across the Full SDLC
*Engineering Enablement — Laura Tacho*

A must-listen (38 min) for anyone trying to make the case for AI investment. AI is now showing up across planning, prototyping, review, and documentation — not just code generation — but AI readiness turns out to be a **developer experience problem**, not a tooling one. The biggest blockers: missing documentation, inconsistent environments, weak CI, and unclear system boundaries. The episode introduces a clean measurement framework: **Amplification** (higher human throughput — measure human output) vs **Augmentation** (agents doing work beyond human capacity — measure agent output vs cost). A critical warning: correlation vs causation makes attribution hard, because high AI usage correlates with already-high performers. The most forward-looking insight: as background agents produce work faster than humans can process it, human workflows become the new bottleneck — and engineers shift toward reviewing, guiding, and orchestrating. Documentation is shifting from optional to foundational infrastructure.

### 3. Six Principles for Designing Human-Agent Interaction
*Every (Thesis) — Karri Saarinen, CEO of Linear*

AI's "slippery feeling" is a design problem, not a model problem — and it almost always traces to the interface. Saarinen has spent a year building agents into Linear and developed six Agent Interaction Guidelines: (1) **Always disclose** the agent is an agent — clear badges, never mistakable as a human. (2) **Inhabit the platform natively** — agents should use the same patterns and actions humans use, making their work legible without a new mental model. (3) **Provide instant feedback** — silence from an agent creates the same anxiety as silence from a colleague. (4) **Be transparent about internal state** — show thinking/waiting/executing/finished; let users inspect reasoning and tools. (5) **Respect requests to disengage** — stop immediately and stay stopped; an agent that keeps going after being told to stop erodes trust faster than one that makes mistakes. (6) **An agent cannot be held accountable** — the human who delegated is. Design the environment well, let the agent run, and own what it produces. The key insight: "The reward for getting this right is that, over time, you can hand agents more and more of the work that doesn't need you."

### 4. The November AI Inflection Point (and What Comes After)
*Simon Willison's Newsletter*

Simon Willison appeared on Lenny's Podcast in a wide-ranging conversation about agentic engineering. The core claim: November 2025 was a genuine inflection point — GPT-5.1 and Claude Opus 4.5 crossed from "code mostly works but you must pay close attention" to "almost all of the time it does what you told it to do," which makes all the difference for autonomous agents. Willison argues software engineers are bellwethers for all information workers: code is easier to verify than legal briefs or essays, but the same disruption is coming for every knowledge profession. The bottleneck has shifted from implementation to **testing and validation** — a spec takes 3 hours now, not 3 weeks, so the question is how to test 3 variants you can now prototype for free. The dark side: "I can fire up four agents in parallel and have them work on four different problems. By 11 AM, I am wiped out for the day." New limits are needed for sustainable AI-augmented work. His prototyping superpower — the ability to show up at a meeting with a working demo — is now commoditised. "Anyone can do what I could do."

### 5. AI Usage Limits Are Investor Narrative Management, Not Capacity Constraints
*Kent Beck — Software Design: Tidy First?*

Kent Beck offers the most cogent analysis of why all three major AI providers cut usage limits simultaneously: it's not chips (different companies have different supply positions), not raw compute (would hit different companies at different times), not economics (they have basically unlimited capital). The real bottleneck is **the investor narrative** — at some point "trust us, it'll work out" stops being enough, and visible limit management signals you're working toward profitability. All three moved together because the same investor class, at the same stage, faced the same "prove it" moment. What breaks the pattern: one company cracks unit economics through distillation, caching, or smarter routing to smaller models — then opens the throttle while competitors can't. For developers, the effect is a deliberate funnel: casual users get free/capped tiers, serious users get pushed to the API. The uncomfortable question: is this a temporary bridge while supply catches up, or the beginning of a new equilibrium where heavy AI use is a premium product?

### 6. The Five Chokepoints We Built Ourselves
*Scott Galloway — No Mercy / No Malice*

A sharp analysis of five critical infrastructure vulnerabilities created by decades of optimising for shareholder value over resilience. (1) **Strait of Hormuz** — Iran's drone swarms (at $20-50k each) facing $82M jets from $13B carriers; the Pentagon's own 2002 war game showed this exact scenario. (2) **SpaceX** — 84% of US launches, 91% of LEO communications satellites, now combining with xAI at $1.25T valuation, filing a $75B IPO. One person, unelected, at the centre of global space, connectivity, AI, and media. (3) **The Oval Office** — 230 years of quietly reassembling the king: trade policy, war powers, the 2001 AUMF. (4) **Cloud** — Three companies own two-thirds of the market; one AWS Virginia outage knocked out Snapchat, Reddit, Fortnite, Coinbase, DoorDash simultaneously. (5) **TSMC/Taiwan** — 97% of high-end chips; potential 10% global GDP hit from a blockade; meaningful US backup capacity not before 2030. "We didn't stumble into these chokepoints, we built them."

### 7. How to Use OpenClaw as an Engineering Leader
*Engineering Leadership — Gregor Ojstersek*

A practical setup guide for OpenClaw (formerly ClawdBot → Moltbot, now acquired by OpenAI, ~350k GitHub stars). Unlike ChatGPT, it's an autonomous assistant that executes tasks, spins up sub-agents for parallel work, and connects to your existing tools. Gregor walks through installation on a dedicated Mac mini: Node.js, `curl -fsSL https://openclaw.ai/install.sh | bash`, then `openclaw onboard --install-daemon`. Connect via Telegram's BotFather for the easiest setup. Recommendation: add at least two API keys (Anthropic + OpenAI or Gemini) to handle rate limits automatically. 46k+ skills available on Clawhub. Security note: don't install on your main machine — use a dedicated device or DigitalOcean droplet. Key engineering leader use cases: automating repetitive admin tasks, running parallel sub-agents on complex projects, connecting to existing engineering tools.
